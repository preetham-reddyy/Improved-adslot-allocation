KDD Cup 2000 Column Statistics

To help you understand the data, we have provided "basic statistics" about the columns (attributes) in the datasets.  There are statistics for each of the 5 data files (clicks, orders, q1_agg, q2_agg, and q3_agg).  All of the statistics are available in a single 29MB WinZip archive that is encrypted with the same password as the data files.  Download the statistics here (17.1M in WinZip). When you extract the archive there will be a subdirectory for each of the 5 data files.  To view the statistics for a specific data file, point your browser to the index.html file in the appropriate subdirectory.

For each column, we provided the name, the type, the number of null values and the number of non-null values.
When the number of distinct values is small (less than 1000), we show the number of distinct values.
For discrete columns, we display the top 20 values.  Ordered columns are displayed in their natural ordering.
For real-valued columns, the range is discretized into 10 equal width bins and a histogram is given.  The min/max/mean and std-dev are also given.
The column "Normalized" shows the percentages of the non-null values in the distribution, which helps when the percentage of nulls is significant.

If you plan to print this file, go ahead and put some paper in the printer; it's about 170 pages.

Important note: All the statistics are computed at the correct granularity level.
Specifically, each attribute is valid at some level.  Some attributes are customer level attributes, some are session level, some are order header level, some are order line level, some at the clickstream request level, etc.  The statistics are computed as if the data is collapsed to that level to avoid incorrect statistics.  For example:

  • Request Query string and Request Referrer are at the clickstream request level.  That's why if you add up the non and non-null values, you'll have 777,480 for each.
  • Cookie first visit, session start login, session first refferer are all at the session level, where there are 234,954 records.
  • The attribute "Which do you wear most frequent" is a customer-level attribute.  All visitors that have not registered are collapsed into an "anonymous" user and statistics are computed for the customers, resulting in 3,727 values.

    • Registered gender is similar, but note that many people have not filled that question, so there are many nulls (most haven't filled it in because the registration form does not ask for it any more).
Notes on columns:
  • Some columns have the value "Choose," which indicates that the user left the answer to the registration form as "choose."

    • In the dataset we gave out, we already mapped this value to unknown (?).
  • Acxiom columns are most of the times only two valued: Null and True.  Acxiom rarely has negative information.  For example, Motorcycle owner is either null or true.  We have mapped users that were found by Acxiom, but whose value is Null to "false," which really means "presumably false." You can always map the falses back to unknowns if you want to do so.  The nulls in the acxiom columns indicate that the person was not found by Acxiom.  Also, the Acxiom "DMA no XXX" (e.g., no mail solicitation) is somewhat confusing because True means you can solicit.  We chose to leave the Acxiom column names and values as defined by Acxiom.
  • The e-mail was stripped by us.  It only reveals Gazelle and the suffix (.com, .gov, etc).
  • All the times are server time, i.e., Eastern-Standard Time (EST).
  • As you can easily notice, some extreme values exist that are noisy (e.g,. birth year is 1839 to 2089).  No attempt was made by us to clean the data nor to override the binning process, which defaults to splitting the range into 10.
  • There are many ID columns.  What are they used for?  They are mostly used by us to do the star join (which you do not see), but IDs are assigned in increasing order, so there is a good time component that may be uncovered when using them.  You can also use them to sort data (e.g., sessions), when you want to split data into a train/test set based on session by time.
